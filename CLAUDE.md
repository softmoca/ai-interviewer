# CLAUDE.md — 프로젝트 작업 지침

이 파일은 Claude Code가 매 세션 참고하는 프로젝트 규칙/맥락이다.

## 프로젝트 개요
- **이름**: AI 면접관 (CS 기술 면접 준비 서비스)
- **목표**: CS 기술 면접 연습 도구 + 취업 포트폴리오
- **핵심**: 질문 DB에서 첫 질문을 꺼내고, 사용자 답변을 LLM에 넘겨 카테고리 질문 풀을
  참고자료로 주입해 **유연한 꼬리질문**을 생성한다(패턴 B). 세션 종료 후 평가 리포트 제공.

## 기술 스택
- 백엔드: **Spring Boot** (Gradle), Spring Data JPA, Spring Security + JWT
- 프론트엔드: **React** (Vite 권장)
- DB: **H2**(개발 초기) → **로컬 Docker PostgreSQL**(주력) → **AWS RDS**(배포)
- LLM: 외부 LLM API (꼬리질문 생성 + 평가). 응답은 JSON으로 받아 파싱.

## 폴더 구조
```
backend/    # Spring Boot
frontend/   # React
docs/       # 기획서, 아키텍처, 프롬프트설계, 결정사항 (설계 기준)
seed/       # 카테고리별 질문 데이터 (os.json ...)
```

## 반드시 지킬 규칙
1. **설계 기준은 `docs/`를 따른다.** 특히 `docs/결정사항.md`(ADR)의 결정을 존중한다.
2. **기능을 추가/변경하면 관련 `docs/` 문서도 함께 갱신한다.** 새 결정은 결정사항.md에 추가.
3. **비밀정보(LLM API 키 등)는 절대 커밋하지 않는다.** `.env` 사용, `.gitignore`로 제외.
4. 카테고리는 코드에 하드코딩하지 않고 **데이터(테이블)로 관리**한다(확장성).
5. LLM 응답은 항상 **구조화(JSON)**로 받아 백엔드에서 검증 후 저장한다.
6. 평가 점수는 **5점 척도**로 단순화한다.
7. LLM 연동은 **인터페이스로 추상화**한다. 특정 SDK/프로바이더에 코드 전체가 묶이지
   않게 하여 나중에 프로바이더를 교체할 수 있게 한다(결정사항 D18).
8. **`docs/code-quality-standards.md`를 코드 작성/리뷰의 구속력 있는 기준으로 삼는다.**
   모든 백엔드 코드는 이 문서(도메인 3대 침범 방지, 4대 가치, 안티패턴 AP-1~AP-10)를
   만족해야 한다. 도메인 모델은 `docs/domain-design.md`, 리뷰는 `docs/clean-review-prompt.md`를
   따른다(결정사항 D20). JPA로 인한 타협은 해당 문서 §2에 명시된 범위로만 허용한다.
   - **범위 주의**: 위 품질/테스트 문서는 **백엔드(OO/JPA) 전용**이다. 프론트엔드(React)는 그대로
     적용하지 않고 **React 관례**(API 계층 분리, 컴포넌트 책임 분리)로 정리한다(결정사항 D29).

## 작업 완료의 정의 (Definition of Done)
> 하나의 작업은 "코드 변경 + 관련 문서 갱신"이 **함께** 끝나야 비로소 '완료'다.
> 코드만 바꾸고 문서를 안 고치면 그 작업은 끝난 게 아니다.

1. **결정 → ADR 기록**: 중요한 기술 결정(프로바이더 선택, 스키마 방향, 라이브러리 채택
   등)을 내리면 반드시 `docs/결정사항.md`에 새 항목(다음 `D` 번호)으로 근거와 함께 남긴다.
2. **구조/스키마 변경 → 아키텍처 갱신**: 프로젝트 구조·엔티티·API·DB 스키마가 바뀌면
   `docs/아키텍처.md`(및 관련 문서)를 **같은 작업 안에서** 갱신한다.
3. **진행 상태 변경 → 마일스톤 갱신**: 로드맵 상태가 바뀌면 `docs/기획서.md`의 마일스톤
   체크박스를 갱신한다.
4. **새 도메인 행위 → 테스트 동반**: 규칙·불변식이 있는 도메인 행위를 추가/변경하면 그
   테스트를 **같은 작업 안에서** 함께 작성한다(사후 백필이 아니라 동반 작성). 무엇을
   테스트하고 무엇은 제외하는지는 `docs/test-strategy.md`를 따른다(결정사항 D23).
5. **요약 보고**: 작업 종료 시 "무엇을(코드) 바꿨고, 어떤 문서를 갱신했는지"를 함께 요약한다.

## 질문 데이터 (seed)
- `seed/*.json` 형식: category / topic / content / difficulty(1~3) / keywords /
  model_answer / source_url / is_opening
- **A안 9개 카테고리 seed 수집 완료**(총 193문항): 운영체제(os) · 자료구조(data-structure) ·
  데이터베이스(database) · 네트워크(network) · 알고리즘(algorithm) · 소프트웨어공학
  (software-engineering) · 컴퓨터구조(computer-architecture) · 디자인패턴(design-pattern) ·
  개발상식(common-sense). 제너릭 로더가 기동 시 전부 적재(통합 테스트로 검증).
- 질문 추출 기준은 `docs/결정사항.md` D17, 적재 규칙은 D24 참고.

## 개발 로드맵 (요약)
- M1: 스키마 확정 + 질문 seed 수집
- M2: 백엔드 뼈대(인증/세션/LLM 연동) + 텍스트 면접 1회 완주
- M3: 평가 리포트 + 프론트 연결 (MVP 완성)
- M4: 음성 입력(STT), B안 카테고리(Java/Web/Spring), 소셜 로그인
