# 코드 품질 기준 (Code Quality Standards)

> 이 문서는 **AI 면접관** 프로젝트에서 코드를 작성/리뷰할 때 따르는 **구속력 있는 기준**이다.
> (참고: jaeyeonling/spring-roomescape-member 의 품질 문서를 우리 스택에 맞게 각색)
> CLAUDE.md 규칙과 함께, 모든 백엔드 코드는 이 문서를 만족해야 한다.

---

## 0. 환원점 (한 줄)

> **"스프링·JPA·LLM 환경에서도 객체지향(도메인)을 지킨다."**

프레임워크(Spring), 영속화(JPA), 외부 지능(LLM API), HTTP는 모두 **바깥의 관심사**다.
비즈니스 규칙은 이것들에 오염되지 않고 **도메인 객체**가 소유한다. 이 문서의 모든 규칙은
이 한 줄로 환원된다.

---

## 1. 도메인 3대 침범 방지

도메인(`com.aiinterviewer.domain.*`)은 다음 셋에게 침범당하지 않는다.

### 1.1 프레임워크 관심사 차단
- 도메인 클래스에 **Spring 어노테이션 금지**: `@Service`, `@Component`, `@Autowired`,
  `@Transactional`, `@RestController` 등.
- 도메인은 Spring 컨테이너를 모른다. `new`로 생성 가능하고, Spring 없이 단위 테스트된다.
- **검증**: `domain/` 하위에 `import org.springframework.*` 0건. (예외는 §5의 JPA/Auditing 타협)

### 1.2 외부 라이브러리 관심사 차단
- **LLM 침범 금지**: 도메인에 `llm.*`(LlmClient, Gemini 등) 타입이 흘러들지 않는다.
  꼬리질문·평가 "요청/응답 파싱"은 애플리케이션 계층의 몫이고, 도메인은 그 **결과(값)**만 받는다.
- JSON 직렬화 어노테이션(`@JsonProperty` 등) 도메인 진입 금지. (직렬화는 DTO/어댑터의 일)
- 도메인은 표준 자바(`java.lang`, `java.time`, `java.util`)와 자체 타입만 사용한다.

### 1.3 인프라 관심사 차단
- 도메인 **비즈니스 메서드에서 `id` 사용 금지**. `id`는 영속화·응답 식별 목적일 뿐,
  "무엇이 같은가/무엇을 할 수 있는가"의 판단 근거가 되면 안 된다(→ AP-7).
- 도메인이 DB 스키마 세부(snake_case 컬럼명, NULL 허용성, 인덱스)에 의존하지 않는다.
- 트랜잭션 경계·커넥션은 도메인의 관심사가 아니다(애플리케이션 계층).

## 2. 우리 프로젝트의 현실적 타협 — JPA·Auditing (정직하게)

참고 저장소는 순수 JDBC라 "도메인에 프레임워크 어노테이션 0건"이 가능했다.
**우리는 Spring Data JPA를 쓴다(결정사항 D8/아키텍처).** 그래서:

- 도메인 엔티티에 `@Entity`, `@Id`, `@Column`, `@ManyToOne`, `@ElementCollection` 등
  **`jakarta.persistence.*` 어노테이션이 존재한다.** 이는 §1.1의 "순수 도메인 격리"와
  **부분적으로 충돌**한다. **이 충돌을 숨기지 않는다.**
- 또한 생성/수정 시각 자동 기록을 위해 `BaseTimeEntity`가
  `org.springframework.data.annotation.@CreatedDate`/`AuditingEntityListener`에 의존한다.
  이는 도메인 계층에 남은 **Spring 흔적**이다.

### 이것을 우리는 이렇게 정리한다 (실질 기준)
1. **JPA 매핑 어노테이션은 "허용된 타협"이다.** JPA 채택의 대가로 받아들이되, 침범을
   **매핑 메타데이터 수준으로 국한**한다. 즉 어노테이션은 붙되, **동작(behavior)은 오염되지 않게** 한다.
2. **그럼에도 비즈니스 로직은 도메인 메서드가 소유한다.** 엔티티는 필드+getter 덩어리가
   아니라, 상태를 바꾸는 규칙을 **자기 메서드**로 표현한다(예: `InterviewSession.complete()`,
   `Evaluation` 점수 1~5 검증). ORM을 쓴다고 빈약한 도메인(AP-2)이 정당화되지 않는다.
3. **JPA 편의를 도메인 훼손과 맞바꾸지 않는다.** `@Setter` 전면 개방 금지, `@Data` 금지,
   양방향 연관관계·Cascade는 꼭 필요할 때만. Lombok은 `@Getter`/`@NoArgsConstructor(PROTECTED)`
   정도의 보조 도구로만 쓰고, 로직 생성 용도로 남용하지 않는다.
4. **`BaseTimeEntity`의 Spring 의존은 인프라 감사(auditing) 한정 타협**으로 격리한다.
   여기서 더 번지지 않게 한다(감사 외 Spring 타입을 도메인에 추가하지 않는다).
5. **DB 전환 대비**: 벤더 종속 쿼리/기능 대신 JPA 표준을 우선한다(H2→PostgreSQL→RDS).

> 요지: "순수함을 100% 지킬 수 없으니 포기한다"가 아니라, **어디까지 타협했고 그 대가로
> 무엇을 지키는지**를 명시한다. 우리가 지키는 것은 **"규칙은 도메인이 소유한다"**이다.

## 3. 4대 가치

### 가치 1 — Clean Code
- 의미 있는 이름으로 의도를 드러낸다. 패턴 접미사(`Manager`, `Handler`, `Util`)로
  도망가지 않고 **업무 언어**로 명명한다(→ domain-design.md 유비쿼터스 언어).
- 함수는 한 가지 일만(SLAP). 주석은 코드로 못 담는 **의도/근거**만.
- 매직값 금지. 5점 척도 경계(1~5), 난이도(1~3) 같은 상수·규칙은 도메인이 검증한다.

### 가치 2 — Clean Architecture (의존성은 안쪽으로)
```
adapter(web/persistence)  →  application(service)  →  domain
        llm(gemini 구현)  →  llm(LlmClient 인터페이스) ←  application
```
- **의존성 규칙**: 안쪽(domain)은 바깥(web/persistence/llm 구현)을 **모른다.**
- Repository는 Spring Data 인터페이스로 두어 구현 교체 가능(OCP).
- **LLM은 `LlmClient` 인터페이스로 역전(DIP).** 애플리케이션은 `GeminiLlmClient`가 아니라
  `LlmClient`에 의존한다. 프로바이더 교체 시 `llm/gemini/`만 바뀐다(결정사항 D18).

### 가치 3 — SOLID (SRP·OCP·DIP 중심)
- **SRP**: 변경 이유가 하나. 컨트롤러=HTTP 번역, 서비스=흐름/트랜잭션, 도메인=규칙,
  리포지토리=영속화. LLM 프롬프트 조립과 응답 검증은 서비스(또는 전용 컴포넌트)의 몫.
- **OCP**: 카테고리는 데이터로 확장(D13). 새 카테고리 추가에 코드 수정 불필요.
- **DIP**: 도메인/서비스는 **추상(LlmClient, Repository 인터페이스)** 에 의존.
  구체 프로바이더·SDK 타입이 상위로 새지 않는다.

### 가치 4 — 도메인 우선
- 비즈니스 규칙은 **도메인 객체 메서드**로 표현한다(Tell, Don't Ask).
- 서비스는 **오케스트레이션**(조회→도메인 호출→저장→DTO 변환)만 담당하고 판단하지 않는다.
- **LLM 응답은 도메인 진입 전 반드시 검증**한다(구조·점수 범위·필수 필드). 검증 통과한
  값만 도메인/DB로 들어간다(CLAUDE.md 규칙 5, 결정사항 D10).
- 도메인 단위 테스트는 Spring/DB/LLM 없이 ms 단위로 돈다.

## 5. 코드 컨벤션 (우리 프로젝트 버전)

1. 테스트는 **AssertJ**(`assertThat()`), 2개 이상 검증은 `assertSoftly`/`assertAll`.
2. 도메인 필드는 가능한 한 불변 지향. 상태 변경은 **의도 드러내는 메서드**로만.
3. **`@Setter`/`@Data`/`@AllArgsConstructor` 남용 금지.** 생성은 정적 팩터리/의도 있는 생성자.
4. 예외는 의미 있는 도메인 예외 또는 구체 표준 예외(`IllegalArgumentException` 등).
5. 엔티티 컬렉션 매핑은 방어적으로(불변 노출, 외부에서 직접 수정 불가).
6. **문서 동기화 의무**: 스키마/구조/결정 변경 시 관련 `docs/`를 같은 작업에서 갱신(DoD, AP-8).

## 6. 프론트엔드(React) 참고

이 문서의 도메인 순수성 규칙은 **백엔드 중심**이다. 프론트는 다음만 준수한다.
- 표현(컴포넌트)과 상태/통신(API 호출) 관심사 분리, 매직 문자열·중복 최소화.
- 서버 응답 타입을 프론트에 정의해 계약을 명시(백엔드 DTO와 정합).
- (React vs Next.js는 프론트 착수 시 결정 — 결정사항 D19.)
