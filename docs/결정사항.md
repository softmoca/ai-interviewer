# 결정사항 (ADR — Architecture Decision Record)

> 프로젝트를 진행하며 내린 주요 결정과 그 근거를 기록한다.
> "왜 이렇게 했나"에 대한 답변 소재이자, 이후 작업(Claude Code 포함)의 기준이 된다.
> 새 결정이 생기면 아래에 계속 추가한다.

## 2026-07-01 — 초기 기획 결정

### D1. 프로젝트 목표는 "연습 도구 + 포트폴리오" 둘 다
- **결정**: 실전 연습용이자 취업 포트폴리오로 동시 활용.
- **근거**: 잘 동작하는 완성품이 곧 좋은 포트폴리오. 두 목표가 상충하지 않음.

### D2. 결과물은 웹 앱
- **근거**: 포트폴리오로 보여주기 가장 좋고 접근성이 높음.

### D3. 핵심 컨셉은 "질문 DB + LLM 꼬리질문" (패턴 B)
- **결정**: 첫 질문은 DB, 꼬리질문은 카테고리 질문 풀을 참고자료로 주입해 LLM이 생성.
- **대안 검토**:
  - 패턴 A(첫 질문만 DB, 꼬리질문 순수 LLM): 통제 어려움.
  - 패턴 C(꼬리질문까지 트리로 DB 저장): 답변 반영 못 해 실제 면접과 거리 멀.
- **근거**: 통제력(품질)과 자연스러움을 동시에 잡음. 포트폴리오로서도 "질문 풀을
  주입해 LLM을 제어했다"가 설명거리가 풍부함.

### D4. 꼬리질문은 "유연 생성" + 완충 규칙
- **결정**: DB를 참고하되 답변에 맞춰 새로 만들어도 됨(범위 밖 허용).
- **완충**: "질문 풀에서 크게 벗어나지 말되, 명백한 오류·모순은 우선 파고든다".
- **근거**: 실제 면접의 자연스러움이 목표. 완전한 통제는 정형적이라 부적합.

### D5. LLM은 자체 모델이 아니라 외부 API 사용
- **근거**: 꼬리질문·평가는 답변 맥락 이해가 필요 → LLM이 유일하게 자연스러움.
  자체 모델 학습은 노력 대비 데모 임팩트 낮고 취준 기간에 리스크 큼. 프롬프트
  엔지니어링이 더 강한 어필 포인트.

### D6. 백엔드는 Spring Boot
- **근거**: 지원 직무(국내 백엔드)와 직결. 채용 공고 대다수가 Java/Spring.
  LLM 호출은 단순 HTTP라 언어 제약 없음.

### D7. 프론트엔드는 React
- **비고**: Next.js 채택 여부는 구현 시 결정.

### D8. DB는 단계적 전환: H2 → Docker PostgreSQL → RDS
- **근거**: JPA로 전환 비용이 낮음. 로컬 Docker에서 운영과 같은 PostgreSQL을 써
  "개발=운영" 일치. 최종 RDS 배포. 컨테이너 경험도 포트폴리오 강점.

### D9. 인증은 Spring Security + JWT (MVP), OAuth2는 2차
- **근거**: 국내 채용 필수 어필 포인트. MVP는 "사용자별 기록 저장"까지만.

### D10. 평가 리포트: 개념별 점수(5점) + 놓친 키워드 + 모범답안
- **근거**: 포트폴리오 임팩트(리포트 스크린샷). 100점 세분화는 LLM 일관성 낮아 5점 척도.

### D11. 답변 입력: MVP 텍스트, 2차 음성(STT)
- **근거**: 핵심 로직은 텍스트 기반. 음성은 STT·녹음 UI 등 난이도가 높아 입력단
  확장으로 뒤에 붙임. "MVP 완성 후 음성 확장" 서사도 좋음.

### D12. 다룰 범위: MVP=A안(핵심 CS), 2차=B안(Java/Web/Spring)
- **A안**: 자료구조/알고리즘/운영체제/네트워크/데이터베이스/디자인패턴
  + 컴퓨터구조/소프트웨어공학/개발상식.
- **B안**: Java/Web/Spring 등.
- **근거**: "CS 기술 면접" 초점 유지. 좁게 깊게 시작 후 확장.

### D13. 카테고리는 데이터로 관리(확장 가능 설계)
- **결정**: 카테고리를 코드 하드코딩이 아닌 `category` 테이블로 관리.
- **근거**: B안 확장 시 스키마 변경 없이 데이터만 추가.

### D14. 질문 소스 3곳 + 역할 분담
- gyoogle: 핵심 CS 메인 소스 / jbee: 입문자 보완 / hoons 블로그: 면접 톤 + 2차 Spring·Java.
- 각 질문에 **출처 URL 기록**.

### D15. 질문 "수집"과 "적재"를 분리
- **결정**: 먼저 구조화 파일(JSON/CSV)로 수집 → 스키마 확정 후 seed 적재.
- **근거**: 수집은 앱 개발과 독립적으로 병행 가능. 스키마만 맞추면 낭비 없음.

### D16. 작업 환경: 기획은 여기서, 구현은 Claude Code
- **결정**: 기획·설계·프롬프트 설계는 대화형(Cowork)에서 확정 → 구현은 Claude Code.
- **근거**: 판단/논의가 필요한 설계는 대화형이 적합, 실제 코딩·파일 작업은 Claude Code가 적합.
- **문서 관리**: `docs/`에 기획서·아키텍처·프롬프트설계·결정사항을 누적하고, 기능
  추가 시 관련 문서도 함께 갱신.

## 2026-07-01 — 질문 추출(1단계) 기준

### D17. 질문 seed는 촌촌하게, 주제(topic) 단위로 관리
- **결정**: 카테고리 하나를 세부 주제(topic)로 나누고, 주제당 3~4개 질문을 추출.
- **필드**: category / topic / content / difficulty(1~3) / keywords / model_answer /
  source_url / is_opening.
- **난이도 기준**: 1=기본 정의, 2=비교·이유, 3=세부 메커니즘.
- **모범답안**: 3~4문장으로 압축 (평가·모범답안 렌더링에 사용).
- **근거**: 질문 풀이 풍부해야 꼬리질문 참고자료(패턴 B)로서 다양성이 확보됨.
  topic 단위 관리는 카테고리 선택 UI 세분화·통계에도 활용 가능.

## 2026-07-01 — 백엔드 뼈대 착수

### D18. LLM 프로바이더는 개발용으로 Google Gemini(gemini-2.5-flash), 단 인터페이스로 추상화
- **결정**: 개발 단계에서는 **Google Gemini API의 무료 티어(`gemini-2.5-flash`)**를 사용한다.
- **필수 제약**: LLM 연동은 반드시 **인터페이스(`LlmClient`)로 추상화**한다. 특정 SDK나
  프로바이더에 코드 전체가 묶이지 않게 하여, 나중에 **Claude / GPT 등 다른 프로바이더로
  교체 가능**하게 한다. 도메인 서비스는 구현체가 아니라 인터페이스에만 의존한다.
- **비밀 관리**: API 키는 `.env`로 관리하고 **절대 커밋되지 않게** 한다(`.gitignore` 제외).
  코드에서는 환경변수(`GEMINI_API_KEY`)로만 주입받는다.
- **근거**: 무료 티어라 개발 초기 비용이 없고, 꼬리질문·평가 품질 검증에 충분하다.
  인터페이스 추상화로 프로바이더 종속을 피해 포트폴리오 관점에서도 "교체 가능한 설계"를
  보여줄 수 있다.

### D19. 프론트엔드는 React + Vite (TypeScript) 채택 — *2026-07-02 확정*
- **결정**: 프론트 착수 시점(M3)에 **React + Vite + TypeScript**로 확정한다.
  (당초 백엔드 착수 시엔 React vs Next.js를 보류했으나, 아래 근거로 Vite 채택.)
- **근거**: 이 서비스는 로그인 뒤 동작하는 **SPA**라 Next.js의 SSR/SEO 이점이 크지 않다.
  Vite는 설정이 가볍고 dev 서버가 빠르며 초보 친화적. TS로 백엔드 응답 계약을 타입으로 고정.
- **버전**: Node 18 환경 호환을 위해 Vite 5 / React 18 / TS 5로 고정(Vite 7은 Node 20+ 요구).

### D20. 코드 품질 기준을 문서화하고 구속력 있는 기준으로 채택
- **결정**: `docs/`에 품질 3종 문서를 도입하고, 이후 모든 백엔드 코드가 이를 **구속력 있게**
  따르도록 한다.
  - `docs/code-quality-standards.md` — 환원점("스프링·JPA·LLM 환경에서도 객체지향을 지킨다")
    + 도메인 3대 침범 방지 + 4대 가치(Clean Code/Architecture·SOLID·도메인 우선)
    + 안티패턴 AP-1~AP-10.
  - `docs/domain-design.md` — 유비쿼터스 언어, Entity/VO/Aggregate 분류, 규칙 소유자, 패키지.
  - `docs/clean-review-prompt.md` — 5인 거장(Uncle Bob/Beck/Fowler/Evans/조영호) 감사 프롬프트.
- **참고 각색**: jaeyeonling/spring-roomescape-member 의 품질 문서를 우리 스택에 맞게 각색.
  참고 저장소는 순수 JDBC라 "도메인에 프레임워크 어노테이션 0건"이 가능했으나, **우리는
  JPA를 쓰므로** 도메인 엔티티에 `jakarta.persistence.*` 어노테이션이 존재한다.
- **핵심 타협(정직하게 명시)**: JPA 매핑 어노테이션과 `BaseTimeEntity`의 Spring Auditing
  의존은 **'허용된 타협'**으로 인정하되, (1)침범을 매핑 메타데이터 수준으로 국한하고
  (2)비즈니스 로직은 반드시 도메인 메서드가 소유하며 (3)`@Setter`/`@Data` 남용을 금지한다.
- **근거**: "완성품이 곧 포트폴리오"(D1). 일관된 품질 기준과 "왜 이렇게 했는가(타협 포함)"의
  설명력은 백엔드 채용에서 강한 어필 포인트다. CLAUDE.md에 상시 규칙으로 연결한다.

## 2026-07-01 — M2 인증(JWT)

### D21. 인증은 Spring Security + JWT(HS256), 무상태 · 액세스 토큰 전용(MVP)
- **결정**: 자체 로그인(이메일/비밀번호) + **JWT 액세스 토큰**으로 인증한다. 서버는
  **무상태(STATELESS)** — 세션을 저장하지 않고 매 요청의 `Authorization: Bearer <token>`을
  검증한다. 서명은 **HS256**, 라이브러리는 **jjwt 0.12.x**.
- **범위(MVP)**: **액세스 토큰만** 발급한다. 리프레시 토큰/로그아웃 블랙리스트/토큰 회전은
  후속(2차)으로 미룬다(과설계 방지 — Kent Beck 규칙4).
- **비밀번호**: BCrypt로 해시 저장. 원문은 도메인에 남기지 않는다.
- **키 관리**: 서명 키는 환경변수 `JWT_SECRET`(운영). application.yml의 기본값은 개발 로컬
  전용이며 실제 비밀이 아니다(운영 커밋 금지 원칙 유지, 결정사항 D18과 동일 기조).
- **공개/보호 경로**: 공개 = `/api/auth/signup`, `/api/auth/login`, `/api/health`,
  `/h2-console/**`(dev). 그 외 전부 인증 필요.
- **근거**: 국내 백엔드 채용 필수 어필 포인트(D9). 무상태 JWT는 확장/배포에 유리하고
  구현이 단순하다.

### D22. 도메인 순수성 유지를 위한 포트-어댑터 계층 도입
- **결정**: 인증 구현 기술(BCrypt, JWT)이 도메인/애플리케이션에 스며들지 않도록 **포트로
  추상화**하고 구현을 adapter에 격리한다(LlmClient/D18과 동일 패턴, DIP).
  - `PasswordEncryptor`(도메인 포트) ← `BCryptPasswordEncryptor`(adapter, Spring Security 위임)
  - `TokenProvider`(애플리케이션 포트) ← `JwtTokenProvider`(adapter, jjwt)
- **레이어 신설**: `application/`(서비스·유스케이스), `adapter/`(web·security)를 도입한다
  (domain-design.md 목표 구조 실체화). 의존성은 `adapter → application → domain`.
- **도메인 우선 준수**: 가입/인증 규칙은 `User.register()`/`User.authenticate()`가 소유하고,
  비밀번호 해시는 getter로 노출하지 않는다(Tell, Don't Ask). `AuthService`는 조회·트랜잭션·
  흐름 조립만 담당(SRP). code-quality-standards.md의 도메인 침범 방지·AP-1/AP-2/AP-4를 만족.
- **근거**: 프로바이더/알고리즘 교체(예: JWT→Paseto, BCrypt→Argon2) 시 adapter만 바꾸면 됨.
  포트폴리오 관점에서 "스프링 환경에서도 객체지향/의존성 역전을 지켰다"는 설명력 확보.

### D23. 테스트 전략 문서화 및 "행위=테스트 동반" 도입
- **결정**: `docs/test-strategy.md`를 신설하고, 테스트 범위/종류/도구/네이밍을 규정한다.

  - **지금 제외(YAGNI)**: 단순 getter·데이터 전용 클래스, Spring Data 기본 CRUD, **아직 생성

  - **종류(우리 규모)**: 도메인 단위(Spring 없이 ms급, 협력자는 Fake) / 통합

- **참고 각색**: jaeyeonling/spring-roomescape-member ADR-0016(Fake/Mock/H2/Real 4분할)을
  참고하되, 순수 JDBC가 아닌 **JPA/Spring 기반 + 작은 규모**에 맞춰 슬라이스를 통합으로
  흡수해 **2축 + 보조**로 축소했다.
- **거버넌스**: CLAUDE.md DoD에 "새 도메인 행위에는 테스트를 동반한다"(항목4)를 추가.
- **백필**: 본 결정과 함께 M1 도메인 행위 테스트를 백필한다 — `InterviewSession`
  상태전이(complete/abandon 불변식), `Evaluation` 점수(1~5) 검증.
- **근거**: 인증 슬라이스는 테스트가 있으나 M1 도메인 행위(D-refactor로 추가된 규칙)는 없어
  회귀 위험이 있었다. 규칙이 있는 곳을 테스트로 고정하고, 이후 세션 API부터는 코드와 테스트를
  함께 작성해 품질을 유지한다.

## 2026-07-01 — M2 세션 API (골격)

### D24. seed 적재는 데이터 주도(제너릭) 로더로 — 카테고리 하드코딩 금지
- **결정**: `SeedDataLoader`(ApplicationRunner)가 클래스패스 `seed/*.json`을 읽어 카테고리/질문을
  적재한다. **카테고리를 코드에 하드코딩하지 않는다**(D13 실체화).
  - 유도 규칙: **파일명(확장자 제외)=슬러그**, JSON의 `category`=카테고리명, `phase`는 현재
    전부 A안이라 `MVP` 기본값, 노출 순서=파일명 정렬 순.
  - **멱등**: 이미 질문이 있으면 건너뛴다.
  - 새 카테고리는 `seed/<slug>.json` **파일 추가만으로** 코드 변경 없이 적재된다.
- **단일 출처**: seed 원본은 저장소 루트 `seed/`에 두고, Gradle `processResources`가 이를
  클래스패스 `seed/`로 복사한다(중복 없음).
- **생성 경로**: 이 로더가 `Category`/`Question`의 생성 경로가 되므로, 두 엔티티에 검증 팩터리
  (`Category.of`/`Question.of`)와 그 단위 테스트를 함께 추가했다(test-strategy.md/D23).
- **근거**: 확장성(카테고리=데이터), 개발 편의(기동 시 바로 사용 가능), 순수성(JSON 파싱
  어노테이션은 로더 DTO에만, 도메인엔 없음).

### D25. 세션 API 골격 설계 (LLM 제외 슬라이스)
- **결정**: 이번 슬라이스는 **세션 생성 → 첫 질문 서빙 → 답변 기록 → 종료/조회**까지만 구현한다.
  LLM 꼬리질문 생성/평가는 다음 슬라이스로 분리(과설계·미구현 의존 회피).
  - 엔드포인트: `POST /api/sessions`, `POST /{id}/answers`, `POST /{id}/complete`, `GET /{id}`.
  - 첫 질문: 선택 카테고리(또는 전체 랜덤) + 난이도 조건의 오프닝 후보 중 무작위 1개.
- **레이어/책임**: 설정 검증(`InterviewSession.start`)·상태 전이·문답 일관성(`QaLog` 팩터리)은
  도메인이 소유. `SessionService`는 조회·트랜잭션·흐름만(SRP).
- **소유권(인가)은 애플리케이션에서**: "요청자=세션 주인" 검증은 `SessionService`에서 `userId`
  비교로 처리한다. 도메인 비즈니스 메서드에서 id 사용을 피하기 위함(code-quality §1.3, AP-7).
- **응답 형태**: 애플리케이션 결과 record(`StartSessionResult` 등)를 웹 응답으로 그대로 반환한다.
  이들은 엔티티가 아니므로 AP-3에 저촉되지 않고, 재구성이 불필요해 요소를 최소화한다(YAGNI).
- **근거**: 실제로 동작하고 테스트 가능한 최소 완결 슬라이스. LLM 키/구현 없이 진행 가능.

## 2026-07-01 — M2 LLM 꼬리질문

### D26. Gemini 꼬리질문 구현 + `.env` 로딩(spring-dotenv) + 키 없을 때 동작
- **결정**: `GeminiLlmClient`가 Gemini `generateContent`를 실제 호출한다. 프롬프트(면접관 지시
  + 카테고리 질문 풀 + 대화 이력, 프롬프트-설계 §2)를 보내고, `responseMimeType=application/json`
  으로 구조화 응답을 받아 파싱한다. 답변 제출 시 `SessionService`가 꼬리질문 **1~2개**를
  생성해 `QaLog.followUp`으로 저장한다(패턴 B).
  - HTTP 클라이언트는 Spring `RestClient`. jjwt처럼 프로바이더 종속 코드는 `llm/gemini`에만.
  - 응답 계약: `{"follow_up_questions":["..","..(선택)"],"reason":"..","within_pool":true}`.
    `FollowUpResult`는 프로바이더 독립 계약이라 직렬화 어노테이션을 두지 않고, Gemini용 파싱
    DTO는 클라이언트 내부에 둔다.
- **`.env` 로딩**: `me.paulschwarz:spring-dotenv`를 도입한다. `backend/.env`의 `GEMINI_API_KEY`가
  Spring 환경으로 자동 주입된다(초보 친화 — 파일 두고 실행만). 파일이 없어도 앱은 정상 기동.
- **키 없을 때(중요)**: 앱은 죽지 않는다. 키가 비면 `GeminiProperties.hasApiKey()==false` →
  LLM 호출 시에만 `LlmNotConfiguredException` → **HTTP 503(LLM_NOT_CONFIGURED)**. 호출/파싱
  실패는 `LlmCallException` → **502(LLM_CALL_FAILED)**. 다른 API(인증·세션 생성/조회 등)는 정상.
- **트랜잭션**: 답변 기록 + 꼬리질문 저장은 한 트랜잭션. LLM 실패 시 함께 롤백되고 명확한
  오류를 반환한다(부분 저장 방지).
- **테스트 분리(D23)**: 키 없이 도는 흐름은 `LlmClient`를 Fake로 대체(`TestLlmConfig`).
  실제 Gemini 호출 검증(`GeminiLlmClientLiveTest`)은 **OS 환경변수 키가 있을 때만** 실행(없으면 skip).
- **근거**: 무료 티어 프로바이더로 실제 동작 확보하되, 키/네트워크 의존을 격리해 CI·협업에서
  안전하게 테스트 가능. 프로바이더 종속은 여전히 `llm/gemini`에만.

## 2026-07-02 — M2 세션 평가 리포트

### D28. 세션 평가(evaluate) 구현 — 완료 세션 → 개념별 5점 리포트
- **결정**: `LlmClient.evaluate`를 Gemini로 구현하고, 완료된 세션의 전체 대화 이력을 넘겨
  개념별 정확성/깊이(1~5) + 놓친 키워드 + 모범답안 + 총평(overall_comment)을 JSON으로 받아
  `Evaluation` 행들로 저장한다(프롬프트-설계 §4). 점수 1~5 검증은 도메인(`Evaluation.of`)이 소유.
  - 엔드포인트: `POST /api/sessions/{id}/evaluation`(멱등 생성), `GET .../evaluation`(조회).
  - **완료된 세션만 평가**(`InterviewSession.isCompleted`). 미완료면 409, 미평가 조회는 404.
  - **멱등**: 이미 평가된 세션은 LLM 재호출 없이 기존 리포트를 반환한다(중복 비용 방지).
  - `overall_comment`는 세션당 한 건(첫 행)에만 저장하고 조회 시 첫 non-null을 사용한다
    (아키텍처 evaluation 규약).
- **소유권 가드 추출**: 평가가 세션 소유권 검증의 두 번째 소비자가 되어, 공용
  `SessionAccessGuard`로 추출하고 `SessionService`도 이를 재사용하도록 리팩터링(중복 제거).
- **지연 로딩 주의**: `Evaluation.missedKeywords`(@ElementCollection)는 트랜잭션 안에서
  복사해 materialize한다. `open-in-view=false`라 응답 직렬화 시점엔 세션이 닫혀 있기 때문.
- **테스트(D23)**: 흐름은 Fake로(키 없이 생성/조회/409/404/403), 실제 Gemini `evaluate`는
  OS 키 있을 때만(`GeminiLlmClientLiveTest`).
- **의의**: 시작→답변→꼬리질문→종료→평가까지 **텍스트 면접 1회 완주**(M2 목표) 달성.

### D29. 프론트엔드 구조 + 개발 연동(API 계층 분리 · Vite 프록시 · 백엔드 CORS)
- **API 계층 분리**: 컴포넌트는 `fetch`를 직접 쓰지 않는다. `src/api/client.ts`(공통 요청·에러·
  토큰 부착) + `src/api/auth.ts`(엔드포인트별 함수) + `src/api/types.ts`(응답 계약)로 격리한다.
  base URL은 환경변수 `VITE_API_BASE_URL`(미설정 시 `/api`).
- **개발 연동**: 기본은 **Vite 프록시** — 프론트(5173)의 `/api`를 백엔드(8080)로 전달해 dev에서
  **CORS가 불필요**. 프론트가 백엔드를 직접 호출할 때(배포/다른 포트)를 대비해 백엔드에
  **CORS 설정**(`CorsConfig`, 허용 오리진 `app.cors.allowed-origins`, 기본 `localhost:5173`,
  Bearer만 쓰므로 `allowCredentials=false`)도 함께 둔다.
- **품질 기준 범위**: `code-quality-standards.md`·`test-strategy.md`는 **백엔드(OO/JPA) 전용**이라
  프론트엔 그대로 적용하지 않는다. 프론트는 React 관례(API 계층 분리, 컴포넌트 책임 분리)
  수준으로 정리하고, 이번 슬라이스 테스트는 최소(스모크)로 둔다.
- **근거**: 컴포넌트-통신 관심사 분리로 유지보수성 확보. 프록시로 개발 마찰을 없애되 CORS로
  배포 유연성 확보.

### D30. JWT 저장 위치는 localStorage(MVP 기본) — 트레이드오프 명시
- **선택지 비교**:
  - **메모리(JS 변수/상태)**: XSS에 가장 안전(영속 토큰 없음). 그러나 새로고침 시 소실 →
    매번 재로그인 필요(UX 나쁨).
  - **localStorage**: 새로고침 유지·구현 단순. 그러나 **XSS로 탈취 가능**(스크립트가 읽음).
  - **httpOnly 쿠키**: JS가 못 읽어 XSS 탈취 방어에 유리. 그러나 **CSRF 대응**과 **백엔드 쿠키
    발급/세션 방식 변경**이 필요(현재 백엔드는 Bearer 토큰을 응답 바디로 반환).
- **결정**: MVP 기본값으로 **localStorage**를 채택한다(백엔드가 Bearer 토큰을 바디로 주는 현
  구조와 정합, UX 유지). 저장 로직은 `tokenStorage` 한 곳에 격리해 이후 교체를 쉽게 한다.
- **XSS 완화**: React 기본 이스케이프 유지, `dangerouslySetInnerHTML` 지양, 신뢰 못할 HTML 미주입.
- **후속(운영)**: 보안 강화 시 **httpOnly 쿠키 + 리프레시 토큰**으로 이전(D21 후속과 함께).
