# 테스트 전략 (Test Strategy)

> **AI 면접관**의 테스트 방침. "무엇을 어떤 방식으로 테스트하고, 무엇은 지금 하지 않는가"를
> 정한다. 판단 기준은 `docs/code-quality-standards.md`, 대상 설계는 `docs/domain-design.md`.
> (참고: jaeyeonling/spring-roomescape-member ADR-0016을 **우리 규모에 맞게 축소** 각색)

---

## 0. 원칙 (한 줄)

> **"규칙(도메인 행위·불변식)과 흐름(통합)은 테스트하고, 단순 데이터는 테스트하지 않는다."**

테스트는 **깨질 수 있는 판단**을 지킨다. getter처럼 깨질 게 없는 코드는 검증하지 않는다(YAGNI).

---

## 1. 무엇을 테스트하는가

1. **도메인 행위 / 불변식** — 상태 전이, 값 범위·필수값 검증, 팩터리 가드 등 **규칙이 있는 곳**.
   - 예: `InterviewSession.complete()/abandon()`(진행중일 때만 종료), `Evaluation` 점수 1~5,
     `User.register()/authenticate()`.
2. **통합 흐름** — HTTP↔애플리케이션↔도메인, 보안(인증/인가), 예외→상태코드 매핑 등 **여러
   조각이 함께 동작해야 맞는지**.
   - 예: 회원가입→로그인→토큰으로 보호 자원 접근, 401/409 케이스(`AuthIntegrationTest`).
3. **독립 인프라 구성요소** — 컨텍스트 없이도 검증 가능한 순수 구성요소.
   - 예: `JwtTokenProvider` 발급→검증 라운드트립, 위조 토큰 거부.

## 2. 무엇을 지금 하지 않는가 (YAGNI · 근거)

- **단순 getter / 데이터 전용 클래스**: 로직 없는 엔티티 필드, record DTO, 매핑 전용 코드.
  깨질 판단이 없어 검증 가치가 없다.
- **Spring Data 기본 CRUD**: `save`/`findById` 등 프레임워크가 보장하는 동작은 테스트하지
  않는다. **커스텀 쿼리/조건**이 생기면 그때 슬라이스 테스트를 추가한다.
- **아직 생성 경로가 없는 엔티티의 검증**: `Category`·`Question`은 현재 **생성 코드(seed 적재
  등)가 없다.** 지금 팩터리/검증 테스트를 미리 만들면 존재하지 않는 코드를 테스트하는
  셈이라 과설계다(Kent Beck 규칙4). **생성 경로가 생기는 커밋에서 테스트를 함께 작성**한다.
- **같은 규칙의 중복 검증**: 한 규칙은 그것을 소유한 계층에서 한 번만 검증한다(도구 중복 회피).

## 3. 테스트 종류 (우리 규모 = 2축 + 보조)

| 종류 | 무엇을 | 방식/도구 | 위치·네이밍 |
|------|--------|-----------|-------------|
| **도메인 단위** | 도메인 행위·불변식 | **Spring 없이** 순수 객체. 협력자는 **Fake**(가벼운 진짜 구현) | `domain/...`, `*Test` |
| **통합** | 요청→DB→응답 전 흐름, 보안, 예외 매핑 | `@SpringBootTest` + `MockMvc` + H2(실 컨텍스트) | `adapter/...`, `*IntegrationTest` |
| (보조) **구성요소 단위** | 독립 인프라 조각 | 컨텍스트 없이 직접 생성해 검증 | `*Test` |

- 도메인 단위 테스트는 **ms 단위**로 돌아야 한다(컨텍스트 로딩 금지). 이게 도메인 순수성의
  실질 이득이다(code-quality §가치4).
- **Fake > Mock**: 도메인 협력자는 목(mock)보다 **가짜 구현**을 선호한다(예: 테스트용
  `PasswordEncryptor`). 슬라이스에서 협력자 격리가 필요할 때만 Mockito(`@MockitoBean`)를 쓴다.
- 참고 저장소는 슬라이스를 `@WebMvcTest`/`@JdbcTest`/RestAssured로 4분할하지만, 우리는 규모가
  작아 **슬라이스를 통합 테스트로 흡수**한다. 계층이 커지면 그때 분리를 도입한다(YAGNI).

## 4. 도구 · 규칙

- **프레임워크**: JUnit 5. **단언**: AssertJ(`assertThat`). 관련 단언 2개 이상은
  `assertSoftly`(또는 `assertThatCode`)로 묶어 한 번에 보고.
- **구조/가독성**: `@Nested` + `@DisplayName`(한국어로 의도 표현). 테스트명은 "무엇이 어떻게"를
  드러낸다.
- **네이밍**: 단위/구성요소 = `XxxTest`, 통합 = `XxxIntegrationTest`.
- **격리**: 통합 테스트는 H2 인메모리(`create-drop`)로 매 실행 독립. 외부 LLM 등은 호출하지 않는다.

## 5. 앞으로 (Definition of Done 연동)

- **새 도메인 행위에는 테스트를 동반한다**(CLAUDE.md DoD). "규칙 추가 + 테스트"가 한 작업이다.
- **세션 API부터는 테스트를 코드와 함께 작성**한다(사후 백필이 아니라 동반 작성).
- 인증 슬라이스(User/JWT/통합)는 구현과 함께 테스트를 두었고, 본 문서 도입과 함께 **M1 도메인
  행위**(세션 상태전이·평가 점수)를 백필한다(결정사항 D23).
