> 최종 업데이트: 2026-07-01
> 상태: 초안 (구현하며 갱신)

## 1. 기술 스택

| 영역 | 선택 | 비고 |
| --- | --- | --- |
| 프론트엔드 | React | (Next.js 여부는 구현 시 결정 가능) |
| 백엔드 | Spring Boot | 국내 백엔드 채용과 직결 |
| 인증 | Spring Security + JWT | MVP는 자체 로그인, 2차에 OAuth2 |
| ORM | Spring Data JPA | DB 종류 전환을 쉽게 하기 위함 |
| DB | H2 → PostgreSQL(Docker) → RDS | 아래 3번 참고 |
| LLM | 외부 LLM API (개발용 Google Gemini `gemini-2.5-flash`) | 꼬리질문 생성 + 답변 평가. `LlmClient` 인터페이스로 추상화(D18) |
| 컨테이너 | Docker (Compose) | 로컬 DB, 배포 준비 |

## 2. 시스템 구성 (개략)

```
[React 프론트]
     |  (REST API, JWT)
     v
[Spring Boot 백엔드]
     |-- 인증/사용자 관리 (Spring Security + JWT)
     |-- 질문 서비스 (카테고리별 질문 풀 조회)
     |-- 면접 세션 서비스 (대화 상태 관리)
     |-- LLM 연동 서비스 --> [외부 LLM API]
     |                         - 꼬리질문 생성 (질문 풀 참고자료 주입)
     |                         - 세션 종료 후 평가 리포트 생성
     v
[Database]
     - 사용자, 카테고리, 질문, 세션, 문답 로그, 평가 결과
```

## 2.5 백엔드 패키지 구조 (구현 반영)

> 2026-07-01 백엔드 뼈대 착수 시점 기준. 도메인 단위(domain-oriented) 패키징.
> 기준 패키지: `com.aiinterviewer`

```
com.aiinterviewer
├── AiInterviewerApplication      # 진입점 (@EnableJpaAuditing, @ConfigurationPropertiesScan)
├── common/
│   ├── BaseTimeEntity            # created/updated 자동 기록 (@MappedSuperclass)
│   ├── DomainGuard               # 생성 전제조건(빈값/범위) 공통 검사
│   └── HealthController          # GET /api/health (부팅 확인용)
├── config/
│   └── SecurityConfig            # 보안 설정 (JWT 무상태 인증 — D21)
├── domain/                       # 규칙의 소유자 (바깥을 모름)
│   ├── category/                 # Category, CategoryPhase(enum), CategoryRepository
│   ├── question/                 # Question, QuestionRepository
│   ├── user/                     # User, UserRepository, PasswordEncryptor(포트)
│   ├── session/                  # InterviewSession, QaLog, SessionStatus/QaRole(enum), *Repository
│   └── evaluation/               # Evaluation, EvaluationRepository
├── application/                  # 유스케이스/오케스트레이션 (D22)
│   ├── auth/                     # AuthService, TokenProvider(포트), LoginResult/MeResult, 예외
│   └── session/                  # SessionService, StartSessionCommand, *Result, 세션 예외 (D25)
├── adapter/                      # 바깥 세계 어댑터 (D22)
│   ├── web/                      # GlobalExceptionHandler
│   │   ├── auth/                 # AuthController, 인증 요청/응답 DTO
│   │   └── session/              # SessionController, 세션 요청 DTO
│   ├── security/                 # JwtTokenProvider, BCryptPasswordEncryptor, JwtAuthenticationFilter, JwtProperties
│   └── seed/                     # SeedDataLoader, SeedQuestion (제너릭 seed 적재 — D24)
└── llm/                          # LLM 연동 추상화 (결정사항 D18)
    ├── LlmClient                 # 인터페이스 (application은 여기에만 의존)
    ├── dto/                      # FollowUpResult, EvaluationResult (구조화 응답 계약)
    └── gemini/                   # GeminiLlmClient(구현), GeminiProperties (교체 시 이 패키지만 대체)
```

- **의존성 방향**: `adapter → application → domain`. 도메인은 바깥을 import 하지 않는다.
- **포트-어댑터(D22)**: 비밀번호 암호화·토큰 발급 등 기술은 포트(`PasswordEncryptor`,
  `TokenProvider`) 뒤에 숨긴다. 구현(BCrypt/JWT)은 `adapter/security/`에만 존재.
- `LlmClient`가 프로바이더 종속을 흡수한다 → Gemini→Claude/GPT 교체 시 `llm/gemini/`만 교체.
- 세션/질문/평가의 서비스·컨트롤러는 이후 M2 단계에서 같은 레이어 규칙으로 추가한다.

## 3. DB 단계 전략

JPA를 쓰므로 DB 교체 시 코드 변경이 거의 없다. 단계적으로 올린다.

1. **개발 초기 → H2 (인메모리)**: 설정 없이 빠른 기능 검증. (가능하면 PostgreSQL
   호환 모드로 두어 문법 차이로 인한 사고 방지)
2. **개발 주력 → 로컬 Docker PostgreSQL**: "개발=운영" 환경 일치. Docker Compose로 관리.
3. **배포 → AWS RDS(PostgreSQL)**: 프리티어로 시작 가능하나 과금 주의.

## 4. DB 스키마 초안

> 실제 컴럼/타입은 구현 시 확정. 아래는 설계 의도를 담은 초안.
> **구현 반영(2026-07-01)**: `user` → 테이블명 `users`(예약어 회피). `question.difficulty`는
> int 1~3. `keywords`/`missed_keywords`는 `@ElementCollection`(별도 테이블). `interview_session`의
> `categories`는 `categoryIds(Set<Long>) + randomAll + questionCount + difficulty`로 구체화.

### category (카테고리 — 데이터로 관리하여 확장 가능)
- `id` (PK)
- `name` (예: 운영체제, 자료구조)
- `slug` (예: os, data-structure)
- `phase` (MVP / EXPANSION) — A안/B안 구분
- `sort_order`

### question (질문 풀 — 첫 질문 + 꼬리질문 참고자료)
- `id` (PK)
- `category_id` (FK → category)
- `topic` (카테고리 내 세부 주제. 예: "세마포어와 뮤텍스") — 주제 단위 묶음 관리
- `content` (질문 본문)
- `difficulty` (예: 1~3 또는 EASY/MID/HARD)
- `keywords` (핵심 키워드 목록 — 평가 시 활용)
- `model_answer` (모범답안, nullable)
- `source_url` (출처 URL — 추적용)
- `is_opening` (첫 질문 후보 여부, 선택적)

### user (사용자)
- `id` (PK), `email`(unique), `password_hash`, `nickname`, `created_at`/`updated_at`
- (2차) `provider`, `provider_id` — 소셜 로그인용
- **구현 반영(D22)**: 테이블명 `users`. `password_hash`는 도메인에서 getter 미노출(캡슐화).
  가입/인증 규칙은 `User.register()`/`User.authenticate()`가 소유하며, 실제 암호화는
  `PasswordEncryptor` 포트에 위임한다.

### interview_session (면접 세션)
- `id` (PK), `user_id` (FK)
- `categories` (선택한 카테고리들 / 랜덤 여부)
- `started_at`, `ended_at`, `status`

### qa_log (세션 내 문답 로그)
- `id` (PK), `session_id` (FK)
- `question_id` (FK, nullable — 꼬리질문은 LLM 생성이라 null 가능)
- `role` (INTERVIEWER / USER)
- `content` (질문 또는 답변 텍스트)
- `is_follow_up` (꼬리질문 여부)
- `seq` (순서)

### evaluation (세션 평가 리포트)
- `id` (PK), `session_id` (FK)
- `concept` (개념/주제 단위)
- `accuracy_score` (1~5), `depth_score` (1~5)
- `missed_keywords`
- `model_answer`
- `overall_comment`

## 4.5 인증/보안 (JWT — M2, 결정사항 D21/D22)

무상태(STATELESS) JWT 인증. 서버는 세션을 저장하지 않고 매 요청의
`Authorization: Bearer <token>`을 검증한다.

### REST API
| 메서드 | 경로 | 공개 | 설명 |
| --- | --- | --- | --- |
| POST | `/api/auth/signup` | 공개 | 회원가입(이메일/비밀번호/닉네임) → 201 |
| POST | `/api/auth/login` | 공개 | 로그인 → `{ accessToken, tokenType: "Bearer", userId, nickname }` |
| GET | `/api/auth/me` | 보호 | 현재 로그인 사용자 조회 |

### 인증 흐름
1. 로그인 성공 시 `JwtTokenProvider`가 액세스 토큰(HS256, subject=userId) 발급.
2. 이후 요청은 `JwtAuthenticationFilter`가 헤더의 Bearer 토큰을 검증 → 인증 주체(userId)를
   SecurityContext에 설정. 토큰이 없거나 유효하지 않으면 인증 미설정.
3. 보호 경로에서 인증이 없으면 `SecurityFilterChain`이 **401** 반환.

### 계층/포트 (D22)
```
adapter/web(AuthController, DTO)  →  application/auth(AuthService)  →  domain/user(User)
adapter/security(JwtTokenProvider)  ─impl→  application/auth(TokenProvider 포트)
adapter/security(BCryptPasswordEncryptor) ─impl→ domain/user(PasswordEncryptor 포트)
```
- 에러 매핑: `GlobalExceptionHandler`가 중복 이메일→409, 인증 실패→401, 검증 실패→400.
- 키: `JWT_SECRET`(env). 토큰 유효기간: `jwt.access-token-validity-seconds`(기본 3600).

## 4.6 면접 세션 (M2 골격 — 결정사항 D25)

세션 생성 → 첫 질문 서빙 → 답변 기록 → 종료/조회. **모두 인증 필요.** LLM 꼬리질문/평가는
다음 슬라이스.

### REST API
| 메서드 | 경로 | 설명 |
| --- | --- | --- |
| POST | `/api/sessions` | 세션 시작(카테고리 슬러그/전체 랜덤/질문 수/난이도) + 첫 질문 서빙 → 201 |
| POST | `/api/sessions/{id}/answers` | 사용자 답변 기록 → 201 |
| POST | `/api/sessions/{id}/complete` | 세션 종료 |
| GET | `/api/sessions/{id}` | 세션 상세(설정·상태 + 대화 이력) |

### 흐름/책임
- **첫 질문 선택**: 선택 카테고리(또는 전체 랜덤) + 난이도 조건의 오프닝 후보 중 무작위 1개.
  질문이 없으면 404. 잘못된 설정(카테고리 미선택 등)은 도메인 검증으로 400.
- **소유권**: 요청자=세션 주인 검증은 `SessionService`(인가 관심사). 아니면 403.
- **상태**: 진행 중 세션만 답변/종료 가능(도메인 불변식). 위반 시 409.
- **에러 매핑**(`GlobalExceptionHandler`): 404(없음/질문없음) · 403(소유 아님) · 409(상태 충돌) · 400(검증).

### seed 적재 파이프라인 (구현 — D24)
```
저장소 seed/*.json  --(Gradle processResources)-->  classpath seed/
   --(SeedDataLoader, 기동 시)-->  Category/Question 적재(멱등, 카테고리 하드코딩 없음)
```

## 5. LLM 연동 흐름

### 5.1 꼬리질문 생성
1. 백엔드가 현재 세션의 대화 이력 + 해당 카테고리의 **질문 풀(참고자료)**을 구성
2. 면접관 프롬프트(→ `프롬프트-설계.md`)와 함께 LLM에 전달
3. LLM이 사용자 답변을 평가해 꼬리질문 1~2개를 생성 (구조화된 형식으로 반환)
4. 백엔드가 파싱해 `qa_log`에 저장 후 프론트로 전달

### 5.2 세션 평가
1. 세션 종료 시 전체 대화 이력을 LLM에 전달
2. "개념별 정확성/깊이(5점), 놓친 키워드, 모범답안"을 **JSON 형식으로** 반환하도록 요청
3. `evaluation`에 저장 후 리포트 화면에 렌더링

> LLM 응답은 반드시 구조화(JSON)로 받아 파싱 안정성을 확보한다. 점수는 5점 척도로
> 단순화하여 일관성을 높인다.

## 6. 질문 데이터 파이프라인

```
[소스 분석/추출]  -->  [구조화 파일(JSON/CSV)]  -->  [seed 적재]  -->  [question 테이블]
 (사람/AI 판단)         seed/ 폴더에 누적           앱 초기화 시
```

- 수집 단계는 앱 개발과 독립적으로 병행 가능.
- 파일 스키마는 `question` 테이블 컴럼과 정합성을 맞춘다 (category, content,
  difficulty, keywords, model_answer, source_url).
