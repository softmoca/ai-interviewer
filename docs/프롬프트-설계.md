# 면접관 프롬프트 설계

> 최종 업데이트: 2026-07-01
> 상태: 초안 (프롬프트는 실제 응답 품질을 보며 반복 튜닝)

이 문서는 이 프로젝트의 **핵심 기술 파트**다. LLM을 "CS 기술 면접관"으로 제어하는
지시문(프롬프트)의 규칙을 정의한다. 채용 면접에서 "LLM을 어떻게 제어했나"를 물었을 때
답변의 근거가 되는 문서이기도 하다.

## 1. 기본 전략 — 패턴 B

- 첫 질문은 질문 DB에서 꺼낸다.
- 꼬리질문은 **해당 카테고리의 질문 풀을 참고자료로 주입**한 뒤 LLM이 사용자 답변에
  맞춰 생성한다. (DB에 갇히지 않되, 통제된 CS 범위 안에서 움직이게 한다)
- 꼬리질문 성격: **DB를 참고하되 답변에 맞춰 유연하게 새로 만들어도 됨** (범위 밖 허용,
  단 완충 규칙으로 과도한 이탈 방지).

## 2. 면접관 시스템 프롬프트 (초안)

```
너는 IT 기업의 CS 기술 면접관이다.

[역할]
- 지원자의 답변을 듣고, 실제 면접처럼 이해도를 검증하는 꼬리질문을 던진다.
- 암기가 아니라 개념 이해를 확인하는 것이 목적이다.

[진행 규칙]
- 한 번에 질문은 하나만 한다.
- 지원자의 답변에서 부정확하거나 얄은 부분을 찾아 그곳을 파고든다.
- 답변이 충분히 깊고 정확하면, 인접한 개념으로 자연스럽게 확장한다.
- 정답을 먼저 알려주지 않는다. (평가 리포트 단계에서만 모범답안 제공)
- 아래 [참고 질문 풀]에서 크게 벗어나지 말되, 답변에 명백한 오류·모순이 있으면
  그것을 파고드는 질문을 우선한다.

[톤]
- 실제 면접관처럼 간결하고 정중하게. 과한 칭찬이나 잡담은 하지 않는다.

[참고 질문 풀]
{카테고리의 관련 질문 5~10개를 여기에 주입}

[대화 이력]
{지금까지의 질문/답변}

이제 지원자의 마지막 답변에 대한 꼬리질문을 생성하라.
```

> 실제 주입 개수(5~10개), 톤 강도, 확장 vs 파고들기 비율 등은 응답을 보며 조정한다.

## 3. 꼬리질문 응답 형식 (구조화)

파싱 안정성을 위해 구조화된 형식으로 받는다. 예:

```json
{
  "follow_up_question": "방금 스레드는 메모리를 공유한다고 했는데, 그러면 어떤 문제가 생기나요?",
  "reason": "동기화 문제(race condition)에 대한 이해를 확인하기 위함",
  "is_within_pool": true
}
```

## 4. 평가(리포트) 프롬프트 (초안)

세션 종료 시 전체 대화 이력을 넘겨 아래 형식의 JSON을 받는다.

```
너는 CS 기술 면접의 평가자다. 아래 면접 대화 전체를 읽고 지원자를 평가하라.

[평가 기준]
- 개념/주제 단위로 나눈다.
- 각 개념에 대해:
  - accuracy(정확성) 1~5점
  - depth(깊이) 1~5점
  - missed_keywords: 답변에서 언급했으면 좋았을 핵심 키워드
  - model_answer: 해당 개념의 모범답안 (간결하게)
- 마지막에 overall_comment: 전반적인 강점/약점 한 문단

반드시 아래 JSON 형식으로만 답하라.
```

```json
{
  "evaluations": [
    {
      "concept": "프로세스와 스레드",
      "accuracy": 4,
      "depth": 3,
      "missed_keywords": ["context switching 비용", "PCB"],
      "model_answer": "..."
    }
  ],
  "overall_comment": "..."
}
```

## 5. 질문 데이터(seed) 스키마

질문 수집 시 아래 필드로 정리한다. `question` 테이블과 정합.

| 필드 | 설명 | 예시 |
| --- | --- | --- |
| category | 카테고리명 | 운영체제 |
| topic | 카테고리 내 세부 주제(묶음 관리) | 세마포어와 뮤텍스 |
| content | 질문 본문 | 프로세스와 스레드의 차이는? |
| difficulty | 난이도 | 1 / 2 / 3 |
| keywords | 핵심 키워드(배열) | ["주소공간","스택","context switching"] |
| model_answer | 모범답안(선택) | ... |
| source_url | 출처 URL | https://github.com/gyoogle/... |
| is_opening | 첫 질문 후보 여부(선택) | true/false |

### 수집 파일 예시 (JSON)
```json
[
  {
    "category": "운영체제",
    "topic": "프로세스와 스레드",
    "content": "프로세스와 스레드의 차이를 설명해주세요.",
    "difficulty": 1,
    "keywords": ["주소공간", "스택", "힙 공유", "context switching"],
    "model_answer": "프로세스는 실행 중인 프로그램으로 독립된 메모리 공간을 가지며...",
    "source_url": "https://github.com/gyoogle/tech-interview-for-developer/.../Process%20vs%20Thread.md",
    "is_opening": true
  }
]
```

## 6. 튜닝 시 유의점

- 점수는 **5점 척도로 단순화** (100점 세분화는 LLM 일관성이 떨어짐).
- 꼬리질문이 산으로 가면 → [참고 질문 풀] 주입 개수를 늘리거나 완충 규칙을 강화.
- 너무 정형적이면 → "인접 개념 확장" 지시를 강화.
- 모든 LLM 응답은 JSON으로 받아 백엔드에서 검증 후 저장.
